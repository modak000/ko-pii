# Annotation Policy — k-pii 검출 정책

> 본 문서는 k-pii 가 어떤 텍스트를 PII 로 판단하는지 (or 안 하는지) 의 *기준* 을 명문화한 것입니다. KDPII (Li Fei et al., IEEE Access 2024) 의 7가지 주석 원칙 + k-pii 자체 도메인 룰을 종합.

## 1. Maximum Length Annotation (최대 길이 원칙)

가능한 한 *가장 긴 의미 단위* 를 단일 span 으로 잡습니다.

```
입력:  서울특별시 종로구 세종대로 209
✓ 검출: "서울특별시 종로구 세종대로 209" (한 덩어리)
✗ 검출: "서울특별시" + "종로구" + "세종대로 209" (분리)
```

**예외**: KDPII 의 *Minimum Unit Annotation* — 표·DB 컬럼 단위로 다뤄야 할 때는 필드별 분리. `tabular.py` 가 컬럼-단위 처리 시 사용.

## 2. 약칭·변형 → 원형 정규화

```
입력           정규화 후
─────────────────────────
서울대         → 서울대학교        (EDUCATION)
기재부         → 기획재정부        (AGENCY)
경영학과       → 경영학           (MAJOR)
洪吉童 (한자)  → 홍길동           (PERSON)
Hong Gildong  → 홍길동           (PERSON, romanization)
```

매핑은 사전 (`UNIVERSITY_ABBREV`, `PROVINCE_ABBREV`, etc.) 에 정의.

## 3. Context-based Annotation (문맥 기반)

같은 토큰이라도 *문맥* 에 따라 검출 여부가 달라집니다.

```
"강남구 출신"    ← ADDRESS 약함 (loose anchor 키워드 없음, 거부)
"강남구 거주"    ← ADDRESS (살/거주 anchor)
"강남구청장"    ← POSITION (직책 컨텍스트)
"강남"           ← 거부 (단독 = 지역명, PII 아님)
```

## 4. Transliteration 처리 규칙

**기본**: 외래어 음역 자체는 PII 검출 대상 아님.
**예외**:
- 생년월일·나이·키·몸무게의 수치 음역 (`서른두 살` ← 32) — 현재 미구현, 숫자만 검출
- 외국인 이름의 한글 음역 (`마이클잭슨`) — PERSON 으로 처리 (외래어 사전 필요)
- 주소·학교명의 한글 음역 — 영문 원본 우선

## 5. 개인 + 조직 모두 포함

```
PERSON  → 개인 이름
EDUCATION/POSITION → 조직 내 식별 가능 정보
ACCOUNT/BUSINESS_REG → 법인·개인사업자 모두
```

조직 자체는 PII 아님 (예: "보건복지부" 는 검출 안 함) 이지만 *조직 내 개인 식별* 에 기여하는 정보는 검출.

## 6. 추정값 / 미래 정보 제외

```
✗ "약 30대 초반"        — 추정값, 검출 안 함
✗ "2030년쯤 결혼 예정"  — 미래 정보
✓ "1988년 1월 1일생"    — 확정된 과거 정보
```

## 7. 시제 기본 = 과거·현재

미래 정보는 *확정된 경우* 만 검출 (예: 출생증명서의 예정 출산일).

## k-pii 추가 정책

### 8. Multi-gate Validation

각 검출기는 단일 형식 매칭이 아닌 **여러 게이트를 통과** 해야 emit:

| 카테고리 | 게이트 |
|---|---|
| RRN | 형식 + 날짜 유효성 + 체크섬 |
| CARD | BIN 화이트리스트 + Luhn |
| EDUCATION | 사전 매칭 (~330개) |
| MAJOR | 사전 매칭 + 접미사 정규화 |
| EMPLOYEE_ID | tight anchor (키워드가 숫자 직전) |
| ADDRESS | (광역+시군구) 조합 검증 또는 강한 keyword anchor |

### 9. 호칭 vs 풀네임 구분

```
✗ "김부장이 협조 안 함"          — 호칭 (성씨 1자 + 직급) = 거부
✓ "부장 김도경 면담"             — 풀네임 + 직급 = PERSON
✓ "강남구 김도윤 의원"           — 3중 매크로 (지역+이름+직책)
```

성씨 1자 + 단음절 직급 (부장/과장/차장/주임/대리/사원/수석/선임/책임/전임) 은 호칭으로 거부. 풀네임 (2-4자) 은 PERSON.

### 10. 위험도 분기 (RiskLevel)

같은 카테고리라도 sub-type 에 따라 위험도 분기:

| 입력 | 라벨 | 위험도 | 사유 |
|---|---|---|---|
| `010-2847-3915` | PHONE | HIGH | 모바일 = 개인 직통 |
| `02-3479-6128` | PHONE | MEDIUM | 유선 = 사업장·대표번호 가능 |
| `070-7864-3920` | PHONE | MEDIUM | VoIP |
| `1988년 1월 1일` | DT_BIRTH | HIGH | RRN 앞 6자리와 동치 |
| `32세` | AGE | INFO | 단독은 5천만 중 70만 → 식별 약함 |

### 11. 키워드 anchor 정책

키워드와 검출 대상의 *거리* 가 카테고리별로 다름:

| 카테고리 | 키워드 거리 | 사유 |
|---|---|---|
| EMPLOYEE_ID | **직전** (콜론/공백) | 일반 문장에서 "사번이 다르다" 같은 FP 방지 |
| MEDICAL_INSURANCE | 25자 윈도우 | "건강보험증 번호: ..." 같은 라벨링 커버 |
| PRESCRIPTION_ID | 18자 윈도우 | 처방전 양식 표준 |
| FAX | 12자 윈도우 | 짧은 라벨 |
| ACCOUNT | 직전 + 은행명 anchor | "신한 110-..." 도 인정 |
| ADDRESS (loose) | 25자 양방향 | "응암동에서 살던" 같은 대화체 |

### 12. KDPII 표준 준식별자 (Quasi-Identifier)

다음 카테고리는 *단독으론 식별 불가* 이지만 결합 시 재식별 위험:

- `DT_BIRTH` (생년월일)
- `EDUCATION` (학력)
- `MAJOR` (전공)
- `POSITION` (직책)
- `AGE`, `HEIGHT`, `WEIGHT` (측정치)
- `EMPLOYEE_ID` (사번)
- `ADDRESS` (주소)
- `PERSON` (이름)
- `IP`, `PHONE`, `EMAIL`, `POSTAL_CODE`, `VEHICLE` (연락처·위치)

→ `analytics/combined_risk.py` + `k_anonymity` 가 quasi-identifier 조합을 자동 평가. 단일 quasi-identifier 만 있으면 LOW, 3개 이상 조합되면 HIGH.

## 거부 사례 모음 (FP 방지)

본 정책에 따라 *의도적으로 거부* 되는 경우들:

```
✗ "김부장이 협조 안 함"               (호칭, PII 아님)
✗ "사무관이 결재"                    (직책 단독, 키워드 없음)
✗ "회의는 2024년 4월 15일"           (날짜, 키워드 없음)
✗ "약 30대 초반"                    (추정값)
✗ "강남구 출신"                      (지역명 단독, 약한 컨텍스트)
✗ "교번 789012"                     (수학 "교차" 의미 충돌, 키워드 제외)
✗ "전송 02-123-4567"                ("전송" = 일반 동사)
✗ "F. 02-123-4567"                  ("F." = Grade F. 충돌)
✗ "0000-0000-0000-0000"             (placeholder)
✗ "1234-1234-1234-1234"             (카드 BIN 첫자리 1 거부)
✗ "표준코드 123456789"              (의약품 한정 단서 부족)
✗ "256.256.256.256"                 (IPv4 옥텟 범위 초과)
✗ "12장 3456"                       (차량 용도 코드 아님)
✗ "291조 9000"                      (한국어 수량 단위)
✗ "120원 3000"                      (화폐 단위)
✗ "우편번호 09999"                  (09 = 시·도 미할당)
✗ "경기도 강남구"                    (광역+기초 조합 실패)
```
