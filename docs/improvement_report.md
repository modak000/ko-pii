> 기반으로 한 룰 정제 작업의 결과를 기록합니다.

## 3. 주요 변경 사항

### 3.1 ACCOUNT — 은행명 anchor (최대 개선)
- 이전: "계좌" 키워드만 anchor
- 이후: 한국 32개 은행명 (신한/농협/카카오뱅크/토스 등) 도 anchor
- 효과: **TP 42 → 653 (15.5x)**

### 3.2 PERSON — 호칭 거부 + common_words 대폭 확장
- "김부장/박과장/이차장님" 같은 (성씨1자 + 직급) 호칭 거부
- 100+ KDPII 빈출 FP 일반어 추가 (호칭/명사/조사 부착형)
- 효과: FP 8,144 → 5,202 (-36%)

### 3.3 AGE — 조사 처리 + 한글 음역
- 이전: "24살이니까" 의 "이"가 lookahead 거부
- 이후: 조사 허용 (은/는/이/가/도/...)
- 한글 음역: "서른두 살", "스물여섯 살" 등 9단위 표기 지원
- 효과: TP 143 → 512 (+258%)

### 3.4 EDUCATION — 약칭 + 초중고
- 이전: "대학교/대학" 정식명만
- 이후:
  - 약칭 "○○대" 패턴 + 사전 검증 (서울대 → 서울대학교)
  - 초중고 패턴 "○○초/○○중/○○고/○○초등학교/..."
- 효과: TP 71 → 279 (3.9x)

### 3.5 MAJOR — 약칭 매핑
- "컴공과 → 컴퓨터공학", "실음과 → 실용음악학", "의대 → 의예과" 등 50+ 매핑
- 효과: TP 210 → 254 (+21%)

### 3.6 POSITION — "님" suffix anchor
- 이전: "직급:/직책:" 키워드 anchor 만
- 이후: "부장님/사장님/팀장님" 같은 호칭도 emit (단, *직급 부분만*)
- 효과: TP 0 → 295 (대화체 케이스 커버)

### 3.7 DT_BIRTH — 인접 풀네임 context
- 이전: 키워드 anchor 없으면 거부
- 이후: 풀네임 (성씨+조사아닌2자+name_final) 인접 시 emit
- 안전망: 토큰 끝이 조사·동사어미·비-이름종성이면 거부
- 효과: TP 356 → 371 (+4%, FP 도 +2)

## 4. 신규 검출기 (이번 작업 누적)

### 4.1 DT_BIRTH (생년월일)
- 패턴 3종: 한국어 / 숫자 / 년생
- 윤년 검증, 1900~현재년도
- 위험도: HIGH (RRN 앞 6자리와 동치)

### 4.2 EDUCATION (학력)
- 사전: 4년제 ~190 + 전문대 ~130 + 약칭 매핑 ~150
- 초중고 패턴 매칭 (사전 없이 suffix 기반)

### 4.3 MAJOR (전공)
- KEDI 학과 분류 ~250개 + 약칭 ~50개
- 접미사 정규화 (학과/학부/전공/학/과)

### 4.4 POSITION (직책)
- titles 사전 200+ 재사용
- 2-mode: 키워드 anchor / 호칭 anchor

### 4.5 AGE / HEIGHT / WEIGHT
- 범위 검증 (0-150세, 50-250cm, 1-300kg)
- AGE 한글 음역 지원

## 5. 여전한 약점

### 5.1 PERSON (F1=0.126)
- FP 5,202건 — KDPII 의 대화체 일반어가 한국 성씨로 시작하는 케이스 다수
- 정책: PERSON FP↓ vs Recall↑ trade-off — 사용자 도메인 (공문서) 우선 유지
- 추가 개선 여지: 더 많은 common_words + 짧은 이름 휴리스틱

### 5.2 ADDRESS (F1=0.165)
- KDPII 의 "강남구" "응암동" 같은 *단편 주소* 가 대다수 FN
- 정책: 공문서 양식 (광역+시군구) 우선 — 대화체 단편은 정책상 거부
- loose pattern 으로 일부 커버 (살던/이사/거주 anchor)

### 5.3 CARD (F1=0.130)
- **k-pii 가 정확히 거부** — KDPII 합성 카드번호가 Luhn 통과 안 함
- 알고리즘 옳음. KDPII 데이터셋 한계.

### 5.4 EDUCATION/MAJOR Recall 부족
- 사전에 없는 작은 대학·학과 다수 미커버
- 개선 여지: KEDI 공시정보 OpenAPI 로 자동 수집 가능

## 6. 보고서 차원의 인사이트

### 6.1 룰 기반 PII 검출의 한계
KDPII 평가에서 명확히 드러난 trade-off:
- **공문서 (정형)**: F1 ≈ 1.0 — 거의 완벽
- **대화체 (자유 한국어)**: F1 ≈ 0.57 — 약 절반

이는 사용자 도메인 (한국 공공) 에 최적화된 의도된 결과.

### 6.2 정책 결정 사례 (KDPII 분석 기반)

| 결정 | 사유 |
|---|---|
| "교번" 제거 | 수학 "교차/교번" 의미 충돌 |
| "F.", "전송" 제거 | "Grade F.", "데이터 전송" 일반어 충돌 |
| "표준코드" 제거 | 일반 산업 표준과 충돌 |
| EMPLOYEE_ID tight anchor | "사번이 다르다" 같은 FP 방지 |
| KCD 검출기 제거 | 의료 문서 양식 표준 없어 정확도 보장 어려움 |
| POSITION 키워드 + "님" suffix only | 대화체 호칭 cover, 공문서 영향 최소 |

### 6.3 데이터셋 한계
- KDPII 합성 카드: Luhn 비통과 → 평가용 가짜
- KDPII 일부 가공된 학교명: 사전 외 작은 대학 다수
- KDPII PERSON 라벨: 외국인 한글명·별명 포함 (k-pii 의도된 범위 외)
